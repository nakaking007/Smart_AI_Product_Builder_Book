# 04 การสร้าง Gem

> **“หยุดพิมพ์คำสั่งเดิมซ้ำ แล้วเปลี่ยนงานที่ชัดเจนให้เป็นผู้ช่วย AI ที่ทดสอบได้”**

**วันที่ตรวจสอบข้อมูลและเมนู:** 18 กรกฎาคม 2569

> **[HERO-04-01] ชื่อภาพ: จากงานซ้ำสู่ Classic Gem**
> **จุดประสงค์:** เปิดบทด้วยภาพผลลัพธ์ที่ผู้เรียนจะสร้าง
> **สิ่งที่ต้องปรากฏ:** ผู้ใช้ Instructions, Knowledge, Preview, Test และ Share
> **คำบรรยายใต้ภาพ:** กระบวนการสร้างผู้ช่วย AI เฉพาะงานตั้งแต่กำหนดหน้าที่จนแชร์อย่างปลอดภัย

## Pain Point

ผู้ใช้ AI จำนวนมากพิมพ์คำสั่งเดิมทุกวัน แต่คำตอบเปลี่ยนตามถ้อยคำ บางคนสร้าง Gem โดยกำหนดเพียงชื่อแล้วรีบแชร์ ทำให้ขอบเขตกว้าง รูปแบบไม่นิ่ง และไฟล์ Knowledge ถูกเปิดให้คนที่ไม่ควรเห็น

บทนี้สอนสร้าง **Classic Gem** ซึ่งเหมาะกับผู้ช่วยสนทนาที่ทำงานซ้ำตาม Instructions และ Knowledge ผู้เรียนจะได้ Gem หนึ่งตัวพร้อม Test Report, Version Log และคู่มือผู้ใช้ ไม่สร้างเมนูหรือความสามารถที่เอกสาร Google ไม่ได้ยืนยัน

## Learning Objectives

- สร้าง Classic Gem จาก Gemini บนเว็บ
- ออกแบบ Name, Description สำหรับผู้ใช้ และ Instructions
- เพิ่ม Knowledge โดยตรวจสิทธิ์และเวอร์ชัน
- เตรียม Starter Library แม้หน้าจอไม่มีช่องเฉพาะ
- ทดสอบ ปรับปรุง แชร์ และเปิดใช้วงกว้างอย่างรับผิดชอบ
- จัดการ Version, Backup และ Export เอกสารประกอบ

## Prerequisite

- บัญชี Google ที่ใช้ Gemini Apps ได้
- คอมพิวเตอร์และเว็บเบราว์เซอร์ปัจจุบัน
- งานซ้ำหนึ่งงานและไฟล์ตัวอย่างที่ไม่มีข้อมูลลับ
- เวลาเรียน 90 นาที และ Mini Project 30 นาที

> **หมายเหตุเรื่องเวอร์ชัน:** หน้าจออาจต่างตามบัญชี ภาษา ภูมิภาค และนโยบาย Workspace ให้เทียบกับหน้าจอจริง คู่มือทางการยืนยันการสร้างผ่าน Gems/New Gem โดยมี Name, Instructions, Knowledge, Preview และ Save ส่วนการแชร์มี Viewer/Editor และ General access ตามสิทธิ์

## Gem คืออะไร

Gem คือเวอร์ชันปรับแต่งของ Gemini สำหรับเป้าหมายหรือแนวทางที่ทำซ้ำได้ Classic Gem ไม่ทำให้โมเดลรู้ทุกเรื่องเพิ่มขึ้น แต่ช่วยเก็บคำสั่งและความรู้ที่ต้องใช้ซ้ำ ทำให้ผู้ใช้ไม่ต้องอธิบายบทบาทและรูปแบบใหม่ทุกครั้ง

> **คำบรรยายใต้ภาพ:** สถาปัตยกรรมของ Gem ที่มีมนุษย์ตรวจและปรับปรุง

## Workflow

~~~text
Problem → Name/Description → Instructions → Knowledge
→ Preview/Test → Improve → Save → Share → Review Version
~~~

| ประเด็น | Chat ปกติ | Classic Gem |
| --- | --- | --- |
| บทบาทและกฎ | ต้องพิมพ์ใหม่ | เก็บใน Instructions |
| เอกสารอ้างอิง | เพิ่มตามบทสนทนา | เพิ่มเป็น Knowledge |
| ความสม่ำเสมอ | ขึ้นกับ Prompt แต่ละครั้ง | ควบคุมได้มากขึ้นเมื่อทดสอบ |

## Step by Step

### STEP 1: สมัครและตรวจสิทธิ์บัญชี

เปิดบัญชี Google ที่จะใช้เรียน ตรวจอายุที่ใช้บริการได้ตามประเทศ และอ่านนโยบายองค์กร หากเป็นบัญชีโรงเรียนหรือที่ทำงานให้ตรวจว่าผู้ดูแลอนุญาต Gemini Apps และการแชร์ Gems อย่าใช้บัญชีส่วนตัวเก็บเอกสารองค์กร

**ตรวจผลงาน:** บันทึกหลักฐานว่าขั้นนี้เสร็จและไม่มีข้อมูลลับ

> **[SCREENSHOT-04-01] ชื่อภาพ: สมัครและตรวจสิทธิ์บัญชี**
> **จุดประสงค์:** แสดงตำแหน่งและผลลัพธ์ของ STEP 1
> **สิ่งที่ต้องปรากฏ:** หน้าบัญชีที่พร้อมใช้และกรอบเตือนเรื่องนโยบาย Workspace
> **คำบรรยายใต้ภาพ:** สมัครและตรวจสิทธิ์บัญชี บนหน้าจอหรือใบงานที่ตรวจสอบแล้ว

### STEP 2: เข้า Gemini

เปิด https://gemini.google.com ลงชื่อเข้าใช้ แล้วตรวจชื่อบัญชีมุมบนของหน้าจอ หากเมนูด้านซ้ายถูกย่อให้กด Open sidebar

**ตรวจผลงาน:** บันทึกหลักฐานว่าขั้นนี้เสร็จและไม่มีข้อมูลลับ

> **[SCREENSHOT-04-02] ชื่อภาพ: เข้า Gemini**
> **จุดประสงค์:** แสดงตำแหน่งและผลลัพธ์ของ STEP 2
> **สิ่งที่ต้องปรากฏ:** หน้า Gemini พร้อมตำแหน่งบัญชีและ Open sidebar
> **คำบรรยายใต้ภาพ:** เข้า Gemini บนหน้าจอหรือใบงานที่ตรวจสอบแล้ว

### STEP 3: เปิด Gems และเลือก New Gem

เลือก Gems หรือ Explore Gems จากแถบซ้าย แล้วกด New Gem บทนี้ใช้ Classic Gem สำหรับผู้ช่วยสนทนาเฉพาะงาน ไม่ใช่ Gems from Google Labs ซึ่งเป็นฟีเจอร์ทดลองสำหรับ mini-app

**ตรวจผลงาน:** บันทึกหลักฐานว่าขั้นนี้เสร็จและไม่มีข้อมูลลับ

> **[SCREENSHOT-04-03] ชื่อภาพ: เปิด Gems และเลือก New Gem**
> **จุดประสงค์:** แสดงตำแหน่งและผลลัพธ์ของ STEP 3
> **สิ่งที่ต้องปรากฏ:** หน้า Gem manager พร้อมปุ่ม New Gem และป้ายแยก Classic Gem
> **คำบรรยายใต้ภาพ:** เปิด Gems และเลือก New Gem บนหน้าจอหรือใบงานที่ตรวจสอบแล้ว

### STEP 4: กำหนด Name

ตั้งชื่อด้วยสูตร บทบาท + งาน เช่น ผู้ช่วยสร้างแผนการสอน ชื่อควรบอกหน้าที่ได้โดยไม่ต้องอ่านคำอธิบาย หลีกเลี่ยงชื่อกว้างอย่าง ผู้ช่วยอัจฉริยะ

**ตรวจผลงาน:** บันทึกหลักฐานว่าขั้นนี้เสร็จและไม่มีข้อมูลลับ

> **[SCREENSHOT-04-04] ชื่อภาพ: กำหนด Name**
> **จุดประสงค์:** แสดงตำแหน่งและผลลัพธ์ของ STEP 4
> **สิ่งที่ต้องปรากฏ:** ช่อง Name พร้อมตัวอย่างชื่อที่ดีและชื่อที่ควรเลี่ยง
> **คำบรรยายใต้ภาพ:** กำหนด Name บนหน้าจอหรือใบงานที่ตรวจสอบแล้ว

### STEP 5: ออกแบบ Description สำหรับผู้ใช้

คู่มือ Google ของ Classic Gem ที่ตรวจในวันที่ระบุยืนยันช่อง Name, Instructions, Knowledge, Preview และ Save แต่ไม่ได้ยืนยันช่อง Description แยกในหน้าสร้างทุกบัญชี ให้เขียนคำอธิบาย 1–2 ประโยคไว้ในเอกสารออกแบบ และใส่ไว้ตอนต้น Instructions เพื่อให้ Gem แนะนำตนเองเมื่อผู้ใช้ทัก เช่น ช่วยครูสร้างร่างแผนการสอนและถามข้อมูลที่ขาด โดยครูต้องตรวจผลก่อนใช้

**ตรวจผลงาน:** บันทึกหลักฐานว่าขั้นนี้เสร็จและไม่มีข้อมูลลับ

> **[SCREENSHOT-04-05] ชื่อภาพ: ออกแบบ Description สำหรับผู้ใช้**
> **จุดประสงค์:** แสดงตำแหน่งและผลลัพธ์ของ STEP 5
> **สิ่งที่ต้องปรากฏ:** กล่อง Description Canvas และหมายเหตุว่าไม่ใช่เมนูแยกที่ยืนยันใน Classic Gem
> **คำบรรยายใต้ภาพ:** ออกแบบ Description สำหรับผู้ใช้ บนหน้าจอหรือใบงานที่ตรวจสอบแล้ว

### STEP 6: เขียน Instructions

ใช้โครง Role, Objective, Input Check, Workflow, Rules และ Output กำหนดให้ถามเมื่อข้อมูลสำคัญขาด ห้ามเดาแหล่งอ้างอิง และระบุรูปแบบคำตอบที่ตรวจได้

**ตรวจผลงาน:** บันทึกหลักฐานว่าขั้นนี้เสร็จและไม่มีข้อมูลลับ

> **[SCREENSHOT-04-06] ชื่อภาพ: เขียน Instructions**
> **จุดประสงค์:** แสดงตำแหน่งและผลลัพธ์ของ STEP 6
> **สิ่งที่ต้องปรากฏ:** ช่อง Instructions ที่แบ่งหกส่วนด้วยสี
> **คำบรรยายใต้ภาพ:** เขียน Instructions บนหน้าจอหรือใบงานที่ตรวจสอบแล้ว

### STEP 7: เพิ่ม Knowledge

ใต้ Knowledge เลือก Add files แล้วเลือก Upload files หรือ Add from Drive ตามบัญชี อัปโหลดเฉพาะเอกสารที่เกี่ยวข้อง มีสิทธิ์ใช้ และเป็นเวอร์ชันล่าสุด ผู้มีสิทธิ์เข้าถึง Gem ที่แชร์อาจดู Instructions และไฟล์ได้ จึงห้ามใส่ความลับ

**ตรวจผลงาน:** บันทึกหลักฐานว่าขั้นนี้เสร็จและไม่มีข้อมูลลับ

> **[SCREENSHOT-04-07] ชื่อภาพ: เพิ่ม Knowledge**
> **จุดประสงค์:** แสดงตำแหน่งและผลลัพธ์ของ STEP 7
> **สิ่งที่ต้องปรากฏ:** ส่วน Knowledge พร้อมรายการไฟล์ เวอร์ชัน และเจ้าของ
> **คำบรรยายใต้ภาพ:** เพิ่ม Knowledge บนหน้าจอหรือใบงานที่ตรวจสอบแล้ว

### STEP 8: เตรียม Conversation Starter Library

Classic Gem ไม่ได้มีช่อง Conversation Starters แยกที่ยืนยันในคู่มือสร้างปัจจุบันทุกบัญชี ให้สร้าง Starter Library 4–6 ข้อไว้ใน Instructions หรือคู่มือผู้ใช้ และสั่งให้ Gem แนะนำเมนูสั้น ๆ เมื่อผู้ใช้ทัก ตัวอย่าง สร้างแผนการสอน, ตรวจแผนเดิม, สร้าง Rubric และออกแบบกิจกรรม

**ตรวจผลงาน:** บันทึกหลักฐานว่าขั้นนี้เสร็จและไม่มีข้อมูลลับ

> **[SCREENSHOT-04-08] ชื่อภาพ: เตรียม Conversation Starter Library**
> **จุดประสงค์:** แสดงตำแหน่งและผลลัพธ์ของ STEP 8
> **สิ่งที่ต้องปรากฏ:** Starter Library พร้อมตัวอย่างคำสั่งเริ่มต้น 4 ข้อ
> **คำบรรยายใต้ภาพ:** เตรียม Conversation Starter Library บนหน้าจอหรือใบงานที่ตรวจสอบแล้ว

### STEP 9: Test ใน Preview

พิมพ์ Prompt ทดสอบในช่อง Preview ด้านขวา ใช้กรณีข้อมูลครบ ข้อมูลขาด นอกขอบเขต การอ้าง Knowledge และคำขอให้ละเลยกฎ บันทึก Expected Result และ Pass/Fail การใช้ Preview ไม่บันทึก Gem อัตโนมัติ

**ตรวจผลงาน:** บันทึกหลักฐานว่าขั้นนี้เสร็จและไม่มีข้อมูลลับ

> **[SCREENSHOT-04-09] ชื่อภาพ: Test ใน Preview**
> **จุดประสงค์:** แสดงตำแหน่งและผลลัพธ์ของ STEP 9
> **สิ่งที่ต้องปรากฏ:** หน้าต่าง Preview และ Test Matrix ห้ากรณี
> **คำบรรยายใต้ภาพ:** Test ใน Preview บนหน้าจอหรือใบงานที่ตรวจสอบแล้ว

### STEP 10: Improve แบบแก้ทีละจุด

หากไม่ถามเมื่อข้อมูลขาดให้แก้ Input Check หากรูปแบบเปลี่ยนให้แก้ Output หากเดาข้อมูลให้แก้ Rules แล้วใช้ Test Case เดิมซ้ำ เปลี่ยนเพียงตัวแปรเดียวต่อรอบเพื่อรู้ว่าสิ่งใดทำให้ดีขึ้น

**ตรวจผลงาน:** บันทึกหลักฐานว่าขั้นนี้เสร็จและไม่มีข้อมูลลับ

> **คำบรรยายใต้ภาพ:** Improve แบบแก้ทีละจุด บนหน้าจอหรือใบงานที่ตรวจสอบแล้ว

### STEP 11: Share อย่างปลอดภัย

กด Share ข้าง Gem เลือกบุคคลหรือ General access ตามสิทธิ์ กำหนด Viewer หรือ Editor อย่างระมัดระวัง Editor แก้และลบ Gem ได้ ส่วน Viewer ยังดู Instructions และไฟล์ที่แชร์ได้ ตรวจสิทธิ์ไฟล์ทุกครั้งก่อน Copy link

**ตรวจผลงาน:** บันทึกหลักฐานว่าขั้นนี้เสร็จและไม่มีข้อมูลลับ

> **คำบรรยายใต้ภาพ:** Share อย่างปลอดภัย บนหน้าจอหรือใบงานที่ตรวจสอบแล้ว

## การจัดการ Version, Backup และ Export

Classic Gem มีการจัดการผ่านบัญชีและพื้นที่ Drive ตามรูปแบบการแชร์ แต่ผู้สร้างควรมีสำเนาที่ควบคุมเองด้วย

1. ส่งออก Instructions เป็น Markdown เช่น `gem_teacher_instructions_v1.1.md`
2. ทำรายการ Knowledge ด้วยชื่อไฟล์ เวอร์ชัน เจ้าของ และวันที่ทบทวน
3. เก็บ Test Report และภาพหลักฐานแยกจากเอกสารลับ
4. บันทึก Share settings และผู้อนุมัติ
5. เมื่อแก้ Gem ให้เพิ่ม Version Log และทดสอบ Regression Cases
6. หากต้องย้อนกลับ ให้นำ Instructions สำรองเวอร์ชันก่อนหน้ากลับมาและทดสอบใหม่

> **[TIMELINE-04-01] ชื่อภาพ: Gem Version Lifecycle**
> **จุดประสงค์:** แสดงการพัฒนาจาก Draft ไป Release
> **สิ่งที่ต้องปรากฏ:** v0.1 Draft → v0.9 Pilot → v1.0 Release → v1.1 Improve → Retire
> **คำบรรยายใต้ภาพ:** วงจรเวอร์ชันที่เชื่อมกับ Test Report และผู้อนุมัติ

- **ข้อควรระวัง:** แก้ทีละจุด

### Prompt 17: ตรวจความสอดคล้อง Output

- **ใช้กับเครื่องมือ:** Gemini
- **เหมาะกับอาชีพ:** ผู้ตรวจคุณภาพ
- **Input ที่ต้องเตรียม:** คำตอบหลายตัวอย่าง

~~~text
เปรียบเทียบคำตอบ 5 ชุดนี้กับ Output Specification=[...] ให้คะแนนความครบ รูปแบบ ภาษา ความถูกต้อง และความสม่ำเสมอ พร้อมรายการจุดที่ต้องทำให้เป็นมาตรฐาน: [คำตอบ]
~~~

- **Output ที่คาดว่าจะได้รับ:** Quality Report
- **ข้อควรระวัง:** อย่าตัดสินข้อเท็จจริงโดยไม่มีแหล่ง

### Prompt 18: สร้างคู่มือผู้ใช้ Gem

- **ใช้กับเครื่องมือ:** Gemini
- **เหมาะกับอาชีพ:** เจ้าของ Gem
- **Input ที่ต้องเตรียม:** Name Description Starter

~~~text
สร้าง Quick Start หนึ่งหน้าสำหรับ Gem ชื่อ=[...] อธิบายทำอะไร ไม่ทำอะไร Input ที่ต้องเตรียม Starter 6 ข้อ วิธีตรวจคำตอบ และช่องทางแจ้งปัญหา
~~~

- **Output ที่คาดว่าจะได้รับ:** คู่มือหนึ่งหน้า
- **ข้อควรระวัง:** อย่าใส่ลิงก์สมมติ

### Prompt 19: สร้าง Version Log

- **ใช้กับเครื่องมือ:** Gemini
- **เหมาะกับอาชีพ:** เจ้าของ Gem
- **Input ที่ต้องเตรียม:** รายการแก้ไข

~~~text
แปลงรายการแก้ไข=[...] เป็น Version Log มี Version, Date, Changed, Reason, Test Result, Owner และ Rollback Note ใช้ Semantic Versioning แบบง่าย
~~~

- **Output ที่คาดว่าจะได้รับ:** ตารางเวอร์ชัน
- **ข้อควรระวัง:** อย่าอ้างผลทดสอบที่ไม่มีหลักฐาน

### Prompt 20: ตรวจความพร้อมก่อน Share

- **ใช้กับเครื่องมือ:** Gemini
- **เหมาะกับอาชีพ:** เจ้าของ Gem

~~~text
ตรวจความพร้อม Classic Gem ก่อน Share จาก Name=[...] Instructions=[...] Knowledge=[...] Test Results=[...] Access=[...] รายงาน Blocker, High Risk, Improvement และ Release Checklist ห้ามสรุปว่าพร้อมหากยังไม่มีหลักฐาน
~~~

- **ข้อควรระวัง:** เจ้าของข้อมูลเป็นผู้อนุมัติสุดท้าย

## Professional Tips

- เริ่มจากงานเดียว ไม่รวมทุกหน้าที่ไว้ใน Gem เดียว
- ใช้เกณฑ์ที่ตรวจได้แทนคำว่า “ตอบให้ดีที่สุด”
- เก็บ Instructions นอกระบบและกำหนดเจ้าของ Knowledge
- ทดสอบด้วยผู้ใช้ที่ไม่ได้ร่วมเขียน Instructions
- เริ่มแชร์แบบ Private หรือวงเล็กก่อน Public
- ผู้มี Viewer access ยังดู Instructions และไฟล์ได้ จึงอย่าใช้การแชร์แทนระบบรักษาความลับ
- NotebookLM notebooks ไม่สามารถใช้เป็น Source ของ shared Gems ตามคู่มือการแชร์ที่ตรวจในวันที่ระบุ

## Common Mistakes

| ข้อผิดพลาด | ผลกระทบ | วิธีแก้ |
| --- | --- | --- |
| สร้าง Gem ก่อนเข้าใจปัญหา | ขอบเขตกว้าง | กรอก Canvas |
| คิดว่ามี Description/Starter field ทุกบัญชี | ผู้เรียนหาเมนูไม่พบ | ใช้ Design Canvas และ Instructions |
| อัปโหลดทุกไฟล์ | คำตอบสับสนและเสี่ยงข้อมูล | คัด Knowledge |
| Preview แล้วไม่กด Save | Gem ไม่บันทึก | กด Save หลังทดสอบ |
| แชร์ Editor โดยไม่จำเป็น | ผู้อื่นแก้หรือลบได้ | ใช้ Viewer |
| เชื่อว่า Public คือ Marketplace เสมอ | อธิบายฟีเจอร์ผิด | ใช้ชื่อ General access ตามหน้าจอ |
| ไม่ทดสอบ Injection | กฎถูกหลีกเลี่ยงง่าย | เพิ่ม Safety Tests |

## Case Study

สถานการณ์จำลอง: โรงเรียนต้องการให้ครูใช้รูปแบบแผนเดียวกัน ทีมสร้าง Gem จากแบบฟอร์มที่ลบข้อมูลจริงแล้ว กำหนด Output กลาง และทดสอบข้อมูลครบ ข้อมูลขาด นอกขอบเขต และขอคัดลอก Knowledge ผลคือครูได้ร่างโครงสร้างเดียวกัน แต่ยังต้องตรวจตัวชี้วัด กิจกรรม และความเหมาะสมกับผู้เรียน บทเรียนคือ Gem ช่วยมาตรฐานกระบวนการ ไม่แทนวิจารณญาณวิชาชีพ

## Workshop: สร้าง Classic Gem ตัวแรก

**เวลา:** 75 นาที

1. 10 นาที — เลือกงานและกรอก Canvas
2. 10 นาที — สมัคร/เข้า Gemini และสร้าง New Gem
3. 20 นาที — เขียน Description, Instructions และ Starter Library
4. 10 นาที — เพิ่ม Knowledge ที่ผ่านการคัด
5. 15 นาที — ทดสอบ Normal 4 ข้อและ Safety 2 ข้อ
6. 10 นาที — Improve, Save, Backup และตั้ง Share แบบวงจำกัด

**เกณฑ์ผ่าน:** Instructions ครบ ผ่านอย่างน้อย 5 จาก 6 Tests ไม่มีข้อมูลลับ และมี Backup

## Mini Project: Gem ผู้ช่วยประจำอาชีพ

ภายใน 30 นาที สร้าง Gem สำหรับครู นักศึกษา ราชการ ธุรกิจ หรือนักพัฒนา ส่งมอบ Canvas, Instructions v1.0, Knowledge Register, Test Report, Quick Start และ Share Plan

## Checklist

- [ ] บัญชีและนโยบายอนุญาต
- [ ] Name ชัดเจน
- [ ] Description ระบุประโยชน์และข้อจำกัด
- [ ] Instructions ครบและไม่ขัดกัน
- [ ] Knowledge มีสิทธิ์ใช้และเป็นปัจจุบัน
- [ ] Starter Library ใช้งานได้
- [ ] ผ่าน Normal Test และ Safety Test
- [ ] กด Save หลัง Preview
- [ ] Backup Instructions และทำ Version Log
- [ ] ตรวจ Viewer/Editor และ General access
- [ ] ไม่อ้าง Marketplace หรือเมนูที่หน้าจอไม่มี
- [ ] มนุษย์ตรวจผลก่อนนำไปใช้จริง

## Summary

ผู้เรียนได้สร้าง Classic Gem ผ่าน 12 ขั้น ตั้งแต่บัญชี Name, Description เชิงออกแบบ Instructions, Knowledge และ Starter Library ไปจนถึง Test, Improve, Share และเปิดใช้วงกว้างอย่างตรงกับหน้าจอจริง พร้อมระบบ Version, Backup และ Export เอกสารประกอบ

## Review Questions

1. เหตุใด Description และ Starter Library จึงต้องอธิบายแยกจากเมนูที่ยืนยันได้
2. Viewer และ Editor ต่างกันอย่างไร
3. Knowledge แบบใดไม่ควรเพิ่ม
4. เหตุใด Preview ต้องตามด้วย Save
5. Public general access ต่างจากการอ้างว่ามี Marketplace อย่างไร
6. เขียน Safety Test สำหรับงานของตนเอง 3 ข้อ
