# Style Guide

## Editorial voice

Write as an experienced, patient product mentor. Be direct, practical, encouraging, and honest about limitations. Never exaggerate what AI or a platform can do.

## Reader level

- Assume no coding background.
- Explain the purpose before the procedure.
- Define a technical term when it first appears.
- Use short examples before introducing complete projects.
- Avoid unnecessary jargon and unexplained abbreviations.

## Chapter pattern

When drafting is authorized, use this general sequence where appropriate:

1. What the reader will build or learn
2. Why it matters
3. Concepts and prerequisites
4. Guided procedure
5. Test and expected result
6. Common problems and fixes
7. Improvement ideas
8. Safety and privacy notes
9. Summary
10. Practice exercise or checklist

## Language and tone

- Prefer active voice and concrete verbs.
- Address the reader as “you.”
- Keep paragraphs focused on one idea.
- Use numbered lists for ordered procedures.
- Use bullets for unordered choices or checks.
- State uncertainty and version limitations explicitly.
- Avoid hype such as “effortless,” “perfect,” or “guaranteed.”

## Markdown conventions

- Use one `#` heading for the chapter title.
- Use `##` for major sections and `###` for subsections.
- Do not skip heading levels.
- Use fenced code blocks with a language identifier.
- Use descriptive link text rather than raw URLs.
- Use tables only when comparison is clearer than prose.
- Add meaningful alt text for every informative image.

## Code and prompt examples

- Keep examples as small as possible while remaining complete.
- Explain where each file belongs and how to run it.
- Never include real secrets, tokens, email addresses, or personal data.
- Use obvious placeholders such as `YOUR_API_KEY` only when required.
- Show expected output or a clear success check.
- Label pseudocode as pseudocode.
- Store substantial reusable samples under `code/` or `prompts/`.

## Accuracy and verification

- Prefer primary documentation for product-specific facts.
- Record research sources under `references/`.
- Verify current UI labels and procedures near publication time.
- State the verification date for version-sensitive instructions.
- Do not call an example tested unless it was actually run.
- Separate confirmed behavior from assumptions or planned behavior.

## Visuals

- Every visual must teach, clarify, or reduce cognitive load.
- Use consistent naming and visual style across diagrams.
- Keep editable source files when available.
- Store assets in the matching `assets/` subdirectory.
- Do not use copyrighted material without documented permission.

## Accessibility and inclusion

- Do not rely on color alone to communicate meaning.
- Use readable contrast and legible type sizes.
- Provide text alternatives for diagrams and screenshots.
- Use inclusive examples and avoid stereotypes.
- Explain keyboard-accessible alternatives when relevant.

## Terminology consistency

Maintain a project glossary during drafting. Use official product names and capitalization, including Google Apps Script, Chrome extension, Custom GPT, NotebookLM, and Codex.
